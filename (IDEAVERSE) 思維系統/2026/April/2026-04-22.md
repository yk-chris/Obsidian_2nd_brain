## 問題本質：PRD 作為「單一文字敘事」先天無法窮舉條件邏輯

你觀察到的痛點非常精準——這不是 PM 失職，而是 **PRD 的媒介形式本身**不適合承載大量 if/then/else 的條件組合 。文字線性敘述去描述多維條件分支，遺漏是必然而非偶然，也就是為什麼業界會另外採用**決策流程表**（針對 if-else-then 形式）來補足 PRD 文字的不足 。\

## 為什麼 PRD 一定會遺漏 if/then/else

- **媒介錯配**：PRD 是敘事文件，但條件邏輯本質是**多維矩陣**（例如：商品類型 × 銷售方式 × 倉儲方式），用段落文字描述 n 維組合會指數級膨脹且容易漏格 。[cloud.tencent](https://cloud.tencent.com/developer/article/1798058 "https://cloud.tencent.com/developer/article/1798058")
- **會議記錄 ≠ 邏輯規格**：PO 在會議中「記錄所有提到的細節」，產出的是時序性筆記，不是結構化的條件覆蓋表；user 核對時要在腦中自行重建邏輯樹，負擔自然極高 。[projectmanager.com](https://projectmanager.com.tw/%E5%B0%88%E6%A1%88%E7%AE%A1%E7%90%86/product-owner/ "https://projectmanager.com.tw/%e5%b0%88%e6%a1%88%e7%ae%a1%e7%90%86/product-owner/")
- **PRD 追求「清晰勝於完美」**：業界共識是 PRD 是「活文件」，不可能一次寫到完美零遺漏，而是要靠版本控制與持續補充 。[lobehub](https://lobehub.com/zh/skills/laidasan-agentskills-writing-prd "https://lobehub.com/zh/skills/laidasan-agentskills-writing-prd")
- **驗收標準本應由 PO 明確定義**，但若驗收條件沒有拆成可枚舉的 case，就無法用來反推 PRD 是否完整 。[yylab](https://www.yylab.tw/product-owner-role-introduction/ "https://www.yylab.tw/product-owner-role-introduction/")

## 解法：不要用文字核對 PRD，改用「結構化工具」反查

與其讓 user 逐字讀 PRD，不如要求 PM/PO 額外產出以下**可窮舉、可勾選**的附件，把「核對」從閱讀變成填表：

|工具|作用|為何能抓出遺漏|
|---|---|---|
|**決策表 / Decision Table**|列出所有輸入條件欄位 × 所有可能值，每一 row 是一個 case|空白 cell 立即暴露未定義的 if/then/else [uml](http://www.uml.org.cn/requirementproject/201105202.asp "http://www.uml.org.cn/requirementproject/201105202.asp")|
|**狀態機圖 (State Diagram)**|畫出物件的所有狀態與轉換觸發條件|漏畫的箭頭 = 漏掉的邏輯分支 [uml](http://www.uml.org.cn/requirementproject/201105202.asp "http://www.uml.org.cn/requirementproject/201105202.asp")|
|**流程圖 + 異常分支**|主流程每個判斷點都必須標出 Yes/No 兩條路|斷掉的支線即為遺漏 [vocus](https://vocus.cc/article/67eab3e1fd89780001b3cd22 "https://vocus.cc/article/67eab3e1fd89780001b3cd22")|
|**Gherkin / Given-When-Then User Story**|每條驗收條件寫成 Given X When Y Then Z|逼 PO 枚舉場景，而非寫散文 [yylab](https://www.yylab.tw/product-owner-role-introduction/ "https://www.yylab.tw/product-owner-role-introduction/")|
|**Acceptance Criteria 清單**|每個 User Story 附上可勾選的驗收項|user 核對時只需逐項打勾，不需重讀 PRD [yylab](https://www.yylab.tw/product-owner-role-introduction/ "https://www.yylab.tw/product-owner-role-introduction/")|
|**邊界值與異常清單**|專門列出 null、超過上限、非法輸入等 edge case|業務規則遺漏、邊界值遺漏是 PRD 常見三大漏點 [uml](http://www.uml.org.cn/requirementproject/201105202.asp "http://www.uml.org.cn/requirementproject/201105202.asp")|

## 給你作為 user 的實務建議

- **反向要求交付物**：在需求階段就要求 PO 提供 **Decision Table 或 Gherkin 驗收條件**，而不是只有 PRD 文字。當 PO 被迫用矩陣思考，遺漏自然減少 。uml+1
- **以「反例」而非「正例」審查**：不要問「這個規格對不對」，改問「當 A=X 且 B=Y 時會怎樣？」—由你出題、PO 填格，能快速暴露未定義區域 。[uml](http://www.uml.org.cn/requirementproject/201105202.asp "http://www.uml.org.cn/requirementproject/201105202.asp")
- **版本控制 + 修訂追蹤表**：PRD 必須附修訂紀錄，每次會議後更新的條件要明確標註，才能追溯「哪些細節是第幾次會議才補上的」，避免口頭共識遺失 。pagerank+1
- **面對面 walkthrough**：在 PRD 定稿前做一次 1:1 邏輯走讀，比文字來回修改更能抓出隱性 if/then/else 。[pagerank](https://pagerank.ing/prd/ "https://pagerank.ing/prd/")
- **接受 PRD 是活文件**：別期待 PRD 一次到位，要期待的是「發現遺漏時，有多快能被補進規格並通知所有下游」的機制 。[lobehub](https://lobehub.com/zh/skills/laidasan-agentskills-writing-prd "https://lobehub.com/zh/skills/laidasan-agentskills-writing-prd")

## 一句話總結根因

你遇到的不是「PO 不認真」，而是**用散文描述條件邏輯**本身就是錯的工具選擇——必須用**決策表、狀態機、Gherkin** 這類結構化產物把 if/then/else 外部化，PRD 才可能被有效核對 。yylab+2

## 決策表 / Decision Table、狀態機圖 / State Diagram、流程圖 + 異常分支

> 簡單來說，這三種工具就像是做系統分析時的三個鏡頭：**決策表看「條件 → 結果」**、**狀態機圖看「狀態 → 轉變」**、**流程圖看「步驟 → 路徑」**。核心在於：它們不是互相取代，而是各自回答不同問題。

---

## 一眼看懂差異

### 1) 決策表 / Decision Table

- **最適合回答：**「在什麼條件下，系統要做什麼決定？」
- **核心重點：** 規則、條件組合、輸出結果
- **最像：** Excel 規則對照表 / if-then-else 的總表
- **常見用途：** 權限判斷、審批規則、狀態切換條件、欄位是否可編輯

### 2) 狀態機圖 / State Diagram

- **最適合回答：**「這個物件現在在哪個狀態？什麼事件會令它轉去下一個狀態？」
- **核心重點：** 生命週期、狀態轉移、觸發事件
- **最像：** 包裹追蹤狀態（已建立 → 審批中 → 已批准 → 已完成 / 已取消）
- **常見用途：** 任務狀態、單據狀態、工單狀態、申請流程狀態

### 3) 流程圖 + 異常分支

- **最適合回答：**「整個流程怎樣走？如果中途出錯或不符合條件，會怎樣處理？」
- **核心重點：** 流程步驟、角色交接、例外處理
- **最像：** SOP 路線圖
- **常見用途：** 業務流程、跨部門交接、系統操作流程、錯誤處理流程

# 1. 決策表 / Decision Table

> 簡單來說，**決策表就像一張「規則總表」**，把不同條件組合與對應結果一次過列清楚，避免漏規則。

## What｜是什麼

決策表是一種把「**條件**」和「**結果／動作**」對應起來的表格工具。  
它的價值在於：當規則很多、條件互相交叉時，用文字描述會很亂，但用表格就可以一眼看到所有組合。

**舉例：**  
例如在 Streamer 類型系統中，你之前提到的邏輯像「當選擇帶有 DIW 標籤的 template 時，只啟用 DIW 任務；所有 DIW 任務完成後才可進入下一步」，這種條件與結果的對照，就很適合先整理成決策表。你的內部材料亦顯示系統中存在大量任務狀態、審批、完成邏輯與條件判斷。

## Who｜誰在用

- Business Analyst：用來釐清業務規則
- Developer：用來寫系統邏輯
- Tester / UAT User：用來檢查是否所有條件組合都有測到
- Process Owner：用來確認規則沒有矛盾

**舉例：**  
在 [新版TRT Streamer平台用戶驗收測試（UAT）啟動會議及功能演示-2026-04-08_18-54-54.pdf](https://haesl-my.sharepoint.com/personal/yk-chris_chan_haesl_com/Documents/Microsoft%20Teams%20Chat%20Files/%e6%96%b0%e7%89%88TRT%20Streamer%e5%b9%b3%e5%8f%b0%e7%94%a8%e6%88%b6%e9%a9%97%e6%94%b6%e6%b8%ac%e8%a9%a6%ef%bc%88UAT%ef%bc%89%e5%95%9f%e5%8b%95%e6%9c%83%e8%ad%b0%e5%8f%8a%e5%8a%9f%e8%83%bd%e6%bc%94%e7%a4%ba-2026-04-08_18-54-54.pdf?web=1&EntityRepresentationId=bb05c019-34c9-4e73-aee6-01bd2254e122) 中，可見與任務狀態、審批流程、父子任務關聯、延遲影響等相關的規則討論，這些都非常適合先收斂成決策表。 
## Why｜為什麼重要

因為系統問題很多時不是「功能沒做」，而是「**規則沒講清楚**」。  
決策表特別能避免以下情況：

1. 少了一個條件組合
2. 兩條規則互相打架
3. 測試只測了常見情況，沒測邊界／例外情況

**舉例：**  
例如「審批者可否直接修改？」、「完成日期可否輸入未來日期？」這類規則若只用文字講，很容易出現理解偏差；若整理成條件表，就會清晰很多。這些問題在你的 UAT 文件中都有明確出現。
## When｜何時適用

當你遇到以下情況時，決策表最有用：

- 有很多 if / else 條件
- 同一功能會因角色、狀態、標籤、日期、權限而改變結果
- 想做 BRD / FRD / UAT 測試案例時

**舉例：**  
像「Task 狀態 = DIW 且 Template 標籤 = DIW 時，哪些欄位可輸入？」這種就是典型決策表場景。

## How｜如何運作

一般做法很簡單：

1. 列出所有**條件**（例如角色、狀態、標籤、日期、審批結果）
2. 列出所有**可能動作／輸出**（例如允許編輯、送審、關閉、顯示警告）
3. 橫向列出各種組合
4. 檢查是否有遺漏或矛盾

**生活化比喻：**  
就像餐廳套餐規則表：

- 堂食 / 外賣
- 是否加配
- 是否會員  
    對應不同價錢與優惠。

## 進階視角

決策表最怕的不是複雜，而是**條件爆炸**。  
如果條件太多，組合數會急增，這時要做：

- 條件分群
- 規則正規化
- 找出 mutually exclusive（互斥）條件
- 拆成多張子決策表

## 心智模型總結句

**簡單來說，決策表就像「系統規則的總開關面板」，其核心在於把所有條件組合和對應結果一次講清楚。**

# 2. 狀態機圖 / State Diagram

> 簡單來說，**狀態機圖就像一個物件的人生地圖**，重點不是做了哪些步驟，而是它現在處於什麼狀態、因什麼事件改變狀態。

## What｜是什麼

狀態機圖描述的是一個對象（例如任務、申請、工單、文件）從開始到結束，會經歷哪些**狀態**，以及在什麼**事件／條件**下轉去另一個狀態。

它不是畫「流程步驟」，而是畫「狀態變化」。

**舉例：**  
在你的 UAT 文件中，任務會有不同狀態，審批請求發出後需要等待批准，批准後狀態才正式更新；這種「事件觸發狀態改變」就是典型狀態機思維。 

## Who｜誰在用

- 系統分析師：定義對象生命週期
- 開發人員：實作狀態轉移規則
- Tester：驗證狀態是否可正確轉移
- 業務用戶：確認流程是否符合實際管理方式

**舉例：**  
如果你要講 Streamer / Task / Approval Request 的狀態如何由 Draft → In Progress → Pending Approval → Approved → Completed / Cancelled，最合適的圖通常是狀態機圖。

## Why｜為什麼重要

因為很多系統 bug，都跟「**狀態轉換不清楚**」有關，例如：

- 本來不應該由 Completed 再跳回 Draft
- Declined 後是否可以重新提交？
- Approved 後是否仍可編輯？
- 某些狀態下哪些欄位應鎖定？

這些問題用流程圖不一定講得清楚，但狀態機圖會很清楚。

**舉例：**  
你的內部材料中提到任務狀態、審批更新、父子任務延遲影響等，正好反映「一個狀態變動會帶來其他行為或限制」。

## When｜何時適用

當你要處理的是「**一個東西的狀態生命週期**」時，用狀態機圖最適合，例如：

- 工單
- 請假申請
- 採購申請
- 任務卡
- 文件簽核

## How｜如何運作

基本元素通常有三個：

1. **State（狀態）**：例如 Draft、Submitted、Approved
2. **Transition（轉移）**：狀態與狀態之間的箭咀
3. **Trigger（觸發）**：例如 submit、approve、reject、close

**生活化比喻：**  
就像包裹追蹤：

- 已下單
- 已出貨
- 運送中
- 已簽收
- 退回

每個狀態之間都要有明確事件觸發。

## 進階視角

進一步做系統設計時，狀態機圖還會涉及：

- entry / exit action（進入／離開狀態時自動做什麼）
- guard condition（守衛條件：符合某條件才可以跳）
- nested state（巢狀狀態）
- parallel state（平行狀態）

## 心智模型總結句

**簡單來說，狀態機圖就像「一個物件的生命週期地圖」，其核心在於說清楚它身處哪個狀態，以及什麼事件會令它轉變。**

# 3. 流程圖 + 異常分支

> 簡單來說，**流程圖是畫正常路線，異常分支是把「出錯時怎麼辦」也畫出來**。

## What｜是什麼

流程圖是用圖形把工作步驟、判斷點、輸入輸出、角色交接畫出來。  
而「異常分支」就是在正常流程以外，把例外情況也畫清楚，例如：

- 資料不齊
- 審批被拒
- 系統失敗
- 權限不足
- 超時未處理

## Who｜誰在用

- 業務團隊：看懂 SOP
- BA / PM：梳理 As-Is / To-Be 流程
- Developer：理解端到端流程
- QA / UAT：設計正常與異常測試場景
- 管理層：看風險點與交接點

**舉例：**  
[New eTRR Application System.pptx](https://haesl.sharepoint.com/sites/PlanningBusinessSupport/TechnicalRecords/_layouts/15/Doc.aspx?sourcedoc=%7B73535623-5356-4DAB-98C4-423EABB82C53%7D&file=New%20eTRR%20Application%20System.pptx&action=edit&mobileredirect=true&DefaultItemOpen=1&EntityRepresentationId=b3b10190-07bb-4a1c-9a0f-2730f339ff3d) 的片段中可見明確的 **Flow Chart**。  
另外，[系統UIUX優化深度研討 - 任務視圖、所有權及日曆功能改進策略-2026-04-10_20-32-36.pdf](https://haesl-my.sharepoint.com/personal/yk-chris_chan_haesl_com/Documents/Microsoft%20Teams%20Chat%20Files/%e7%b3%bb%e7%b5%b1UIUX%e5%84%aa%e5%8c%96%e6%b7%b1%e5%ba%a6%e7%a0%94%e8%a8%8e%20-%20%e4%bb%bb%e5%8b%99%e8%a6%96%e5%9c%96%e3%80%81%e6%89%80%e6%9c%89%e6%ac%8a%e5%8f%8a%e6%97%a5%e6%9b%86%e5%8a%9f%e8%83%bd%e6%94%b9%e9%80%b2%e7%ad%96%e7%95%a5-2026-04-10_20-32-36.pdf?web=1&EntityRepresentationId=11246801-cdb3-4fd2-a3db-e4198908ab2e) 亦提到多個流程痛點，例如篩選失效、操作後自動跳回頂部、module task 顯示不全等，這些都很適合作為流程圖的異常分支。 
## Why｜為什麼重要

因為現實世界的流程，永遠不只「happy path（理想路徑）」。 如果只畫正常流程，最後通常會出現三種問題：

1. 真正出錯時沒人知道怎樣處理
2. 部門之間責任不清
3. 系統邏輯只考慮成功，沒考慮失敗

## When｜何時適用

以下情況特別適合用流程圖 + 異常分支：

- 跨部門流程
- 人機互動流程
- 需要交接、審批、通知、補件、退回的流程
- 想做 SOP、BRD、UAT scenario、操作說明書

## How｜如何運作

一般會先畫：

1. **正常主流程**（Happy Path）
2. 再在每個關鍵節點問一句：  
    **「如果失敗／不符合條件／資料不齊，會怎樣？」**
3. 把例外情況分支出來
4. 清楚標明：
    - 誰負責處理
    - 回到哪一步
    - 是否發通知
    - 是否中止流程

**生活化比喻：**  
就像機場登機流程：

- 正常：報到 → 安檢 → 登機
- 異常：
    - 文件不齊 → 補文件
    - 超重 → 付款 / 重新整理
    - 登機閘口更改 → 系統通知 → 改去新閘口

## 進階視角

成熟一點的流程圖，不只畫步驟，還會加：

- Swimlane（角色泳道）
- SLA / 時限
- 系統通知點
- 手動 vs 自動動作
- 失敗後 rollback（回退）策略

## 心智模型總結句

**簡單來說，流程圖 + 異常分支就像「工作地圖 + 意外應變圖」，其核心在於不只告訴你平時怎樣走，也告訴你出事時怎樣處理。**

---

# 三者如何配合使用（最實務的做法）

如果你是在做 **BRD / Functional Requirement / UAT**，最實用的組合通常是：

1. **先畫流程圖 + 異常分支**  
    先看整體流程怎樣走、哪裡有例外
    
2. **再畫狀態機圖**  
    釐清重要物件（例如 task / request / streamer）的狀態生命週期
    
3. **最後補決策表**  
    把所有條件判斷、權限、欄位控制、狀態切換規則收斂成清楚規則表
    

---

# 常見誤解澄清

- **誤解 1：** 決策表 = 流程圖  
    **澄清：** 不是。決策表重點是「條件與結果」，流程圖重點是「步驟與路徑」。
    
- **誤解 2：** 狀態機圖 = 流程圖  
    **澄清：** 不是。狀態機圖重點是「物件當下狀態與轉換」，流程圖重點是「做事順序」。
    
- **誤解 3：** 異常分支可後補，不重要  
    **澄清：** 其實很多 bug、責任不清和 UAT 爭議，正正就是因為異常分支沒先畫出來。
